## 1. 設計方針

### 1.1 アーキテクチャ全体像
要件定義 P-9 (Web アプリ) / N-10 (マルチデバイス) を満たす **SPA + BFF 型**構成。既存スターター (FastAPI + Vue3) の規約をそのまま踏襲する。

```
[ブラウザ SPA]  Vue3 + Vite + TS
   │  src/services/* (型付き API クライアント)
   ▼  HTTPS / JSON  (N-4 TLS)
[BFF / API]  FastAPI  app/api/*  (/api 一括 prefix)
   │  app/services/* (ドメインサービス: 仕訳生成・税計算・帳簿集計)
   ▼
[ドメイン層]  app/domain/*  (Account/Journal/FixedAsset … 複式簿記の不変条件)
   ▼  リポジトリIF
[永続化層]  app/repositories/*  → SQLModel + SQLite/PostgreSQL  (add-persistence)
   │
   └─ 証憑ストレージ (ファイル: 電帳法 F-402/N-5) ・ 監査ログ (N-7)
```

### 1.2 レイヤ責務
| レイヤ | 責務 | 該当 |
|--------|------|------|
| API (`app/api/`) | HTTP I/F・入出力スキーマ(`response_model`必須)・認証 | F-101等の入口 |
| サービス (`app/services/`) | ユースケース。**仕訳の自動生成**・税計算・帳簿集計のロジック | 簡易入力→仕訳化, 減価償却計上 等 |
| ドメイン (`app/domain/`) | エンティティと**不変条件**(借方=貸方, 残高計算) | 複式簿記の中核 |
| リポジトリ (`app/repositories/`) | 永続化の抽象。当面 in-memory → DB へ差替 | N-2/N-3 |

### 1.3 中核となる設計判断
1. **すべての取引は「仕訳」に正規化する** (B-1)。売上・経費・減価償却・決算整理は、各機能サービスが最終的に `JournalEntry` を生成する。帳簿・レポート・申告集計はこの単一の仕訳ストアからの**読み取り専用ビュー**として実装する。これにより機能間の結合を「仕訳を作る側」と「仕訳を読む側」に分離でき、**並列開発が成立する**。
2. **税区分は金額に常に随伴する** (P-3, F-204)。仕訳明細に税区分・税額を保持し、消費税集計(簡易課税)は集計時に再計算せず明細値を積み上げる。
3. **訂正は論理的に追跡する** (F-103, N-7)。物理削除せず訂正履歴(操作ログ)を残し、電帳法の訂正削除要件に配慮する。

## 4. 共通データモデル (複式簿記コア)

全モジュール共通の中核モデル。各機能はこのモデルへ仕訳を**書く**か、ここから**読む**。

```
AccountType (enum)   : ASSET 資産 / LIABILITY 負債 / EQUITY 純資産 / REVENUE 収益 / EXPENSE 費用
TaxCategory          : code, 名称, 種別(課税10/軽減8/非課税/対象外), 税率, 事業区分(簡易課税)
Account 勘定科目      : code, 名称, type, 既定税区分, 表示順, 補助科目[]
Counterparty 取引先   : id, 名称, 登録番号(インボイス)
JournalEntry 仕訳     : id, 日付, 摘要, 会計年度, ステータス, 由来(機能/証憑), 登録/訂正日時
  └ JournalLine 明細  : side(借方/貸方), account, 補助科目, 金額, 税区分, 税額
        ─ 不変条件: Σ借方金額 == Σ貸方金額 (借貸平均の原理)
FixedAsset 固定資産   : 資産名, 取得日, 取得価額, 耐用年数, 償却方法, 未償却残高
Voucher 証憑         : ファイル, 取引日, 金額, 取引先, 紐づく仕訳ID  (電帳法検索キー)
OpeningBalance 期首残高: 年度, 科目, 金額
AuditLog 監査ログ     : 操作種別, 対象, 操作者, 日時, 前値/後値
```

### 4.1 勘定残高の導出
科目残高 = Σ(借方明細) − Σ(貸方明細)。資産・費用は借方が正、負債・純資産・収益は貸方が正(`AccountType` で符号を解釈)。PL/BS/試算表(M7)はこの集計を期間・年度でフィルタした読み取りビュー。

### 4.2 税計算ルール (確定: 簡易課税 / 第五種50%)
- 売上仕訳の貸方(収益)明細に課税区分(10%/軽減8%)を付す。
- 消費税納付額(M8-F804) = Σ課税売上に係る消費税額 × (1 − みなし仕入率50%)。複数事業区分対応の余地を税区分マスタの「事業区分」で残す(検討事項に対応)。

## 5. API 設計規約 (全モジュール共通)

- ルーターは `app/api/<module>.py`、`app/api/__init__.py` で集約。`/api` prefix は `main.py` 一括(個別に付けない)。
- すべて `response_model` 指定。リソース命名は複数形・kebab/スネークは既存に合わせ snake。
- 主要エンドポイント設計(抜粋):

| メソッド | パス | 機能 | M |
|---------|------|------|---|
| GET | `/api/accounts` | 勘定科目一覧 | M1 |
| GET/POST | `/api/tax-categories` | 税区分 | M1 |
| GET/POST | `/api/journal-entries` | 仕訳 取得/登録(バランス検証) | M2 |
| PUT/DELETE | `/api/journal-entries/{id}` | 訂正(履歴保持) | M2 |
| POST | `/api/cash-book/entries` | 出納帳簡易入力→仕訳化 | M5 |
| GET | `/api/reports/trial-balance` | 合計残高試算表 | M7 |
| GET | `/api/reports/pl` `/bs` | 損益計算書/貸借対照表 | M7 |
| POST | `/api/fixed-assets/{id}/depreciation` | 減価償却計上 | M6 |
| GET | `/api/tax/consumption` | 消費税(簡易課税)集計 | M8 |

frontend は `src/services/<module>.ts` に集約(コンポーネントから直接 fetch しない)。

## 6. 並列開発計画 (タスク分割と依存DAG)

機能を「**仕訳を作る側 / 読む側**」に分離した結果、基盤確立後はモジュールを並列開発できる。

```
Layer 0 (逐次・先行): 共通基盤
  M1 core-masters ─┐
  M2 journal ──────┴─→ 仕訳ストア + バランス検証 + 勘定科目/税区分  ← ここまでが全機能の前提

Layer 1 (並列): 仕訳を「作る」機能群 (M2の仕訳APIに依存)
  ┌── M3 receivables (売上→売掛/入金)
  ├── M4 expenses    (経費→未払/支払)
  ├── M5 cash-bank   (出納帳/CSV取込→仕訳候補)
  └── M6 fixed-assets(減価償却→仕訳)

Layer 2 (並列): 仕訳を「読む」機能 (仕訳ストアに依存、Layer1と並行可)
  └── M7 reports (試算表/PL/BS/元帳/ダッシュボード)

Layer 3 (集約): M6+M7 の上に
  └── M8 closing-tax (決算整理・青色申告決算書・消費税集計)

横断 (全Layerに並行): X1 認証 / X2 監査ログ / X3 証憑保存 / M9 data-io / X4 store
```

### 6.1 並列化の根拠 (結合点の限定)
各 Layer1 モジュールは「ドメインイベント→仕訳生成サービス」という同一契約 (`POST /journal-entries` 相当の内部API) のみに依存し、互いを参照しない。M7 は仕訳の**読み取りスキーマ**のみに依存。したがって M3〜M6, M7 は別担当(別エージェント/別ブランチ)で同時並行に開発し、最後に仕訳ストア契約で結合できる。

### 6.2 各モジュールの開発フロー (要件定義 第11章 / CONTRIBUTING に準拠)
1モジュール = `docs/accounting/<機能>/` に **01要件 → 02仕様 → 03基本設計 → 実装 → テスト** を生成 (`write-dev-docs` / `run-dev-cycle`)。実装は `add-fullstack-feature` 系、検証は `verifier-webapp`、確定で `push-changes`。

### 6.3 マイルストーン (要件定義 Phase 対応)
| Phase | 内容 | 含むモジュール |
|-------|------|--------------|
| 1 記帳基盤 | 本書の Layer0 + 出納帳 + 基本帳簿 | M1, M2, M5(出納帳), M7(試算表/元帳) |
| 2 決算機能 | 固定資産・PL/BS・青色申告決算書 | M6, M7(PL/BS), M8(決算書) |
| 3 効率化 | CSV取込・自動仕訳・ダッシュボード | M5(取込), M2(自動仕訳), M7(ダッシュ) |
| 4 法令・連携 | 消費税集計・証憑保存・e-Tax/税理士出力 | M8(消費税), X3, M9 |

## 7. 横断的関心事

| 関心事 | 設計 |
|--------|------|
| 認証 (X1) | 単一ユーザー前提だが N-4 によりログイン必須。`add-auth` の JWT。税理士共有はデータ出力で代替(P-8) |
| 監査ログ (X2) | サービス層の mutation を共通デコレータ/依存で記録。訂正・削除は前後値を保持 |
| 証憑・電帳法 (X3) | ファイル実体 + メタ(取引日・金額・取引先)を保存し、3キー検索を提供。仕訳に添付 |
| エラー/ログ | `add-observability` の構造化ログ。本番は汎用エラーメッセージ |
| 会計年度 | 暦年固定(P-7)。年度繰越(M9-F904)で期末残高→翌期首残高を生成 |

## 9. 残課題 (要件定義 第12章「今後の検討事項」を継承)
- 簡易課税の事業区分が単一(第五種)か複数か → 税区分マスタの事業区分で複数対応の余地を確保済み。確定後に M8 集計を分岐。
- 電帳法の証憑検索要件の実装方式(取引日/金額/取引先) → X3 の検索インデックス設計で詳細化。
- 銀行/カード CSV のフォーマット差異の対応範囲 → M5 でアダプタ方式(金融機関別パーサ)。
- e-Tax 連携の要否 → 当面 M9 の CSV/PDF 出力で代替(任意)。
