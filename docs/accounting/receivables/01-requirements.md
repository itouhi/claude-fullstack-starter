# 要求事項定義書 — receivables (売上・売掛管理)

## 1. 背景・目的

個人事業主がサービス業として役務を提供した際、役務提供時点で売上を認識し(発生主義)、後日入金が確認できたら売掛金を消し込む一連の流れを管理する。現状、エクセルや紙台帳で管理している事業者が多く、未入金の把握漏れや二重入力が発生しやすい。本モジュールは仕訳ベースで売掛金の発生・消込を一元管理し、取引先別の未入金残高をリアルタイムに把握できるようにする。

由来: 要件定義書 F-301 (売上計上) / F-302 (売掛・入金消込) / P-5 (発生主義)

## 2. スコープ

- **対象**:
  - 役務提供に基づく売上の発生主義計上 (売掛金の起票)
  - 入金の確認と売掛金消込 (入金仕訳の起票)
  - 取引先別の未消込売掛残高の照会
- **対象外**:
  - 請求書の発行・印刷・送付 (P-6 により本システムはスコープ外)
  - 商品の販売・仕入・在庫管理 (サービス業前提、要件定義第10章)
  - 分割払い・手形の管理 (TBD: 今後の検討事項)
  - 外貨建て取引 (円建てのみ対応、B-2)

## 3. ステークホルダー

| 役割 | 関心事 |
|------|--------|
| 個人事業主 (主ユーザー) | 誰からいくら受け取っていないかを即座に把握したい。難しい簿記知識なしで入力できること |
| 税理士・記帳担当 | 発生主義で正しく売上が計上されているか。仕訳が標準科目(売掛金135/売上高500)と整合していること |

## 4. 要求事項一覧

## 5. 制約条件

- 金額は円単位の整数で管理する。小数・外貨は扱わない (B-2 / 全体基本設計 §4)
- 勘定科目コードは標準科目マスタの固定コードを使用する: 売掛金=135, 売上高=500, 普通預金=102, 現金=101
- 税区分コードは `T10`(課税売上10%)を売上に適用する (全体基本設計 §4.2)
- 仕訳の生成・保存は共通の仕訳エンジン (M2 `journal`) の `POST /api/journal-entries` 契約を利用する
- バックエンド: FastAPI (Python)。フロントエンド: Vue 3 + TypeScript (Composition API + `<script setup>`)
- 請求書発行機能は本モジュールの対象外 (P-6)

## 6. 前提・リスク

- **前提**:
  - M1 `core-masters` が勘定科目マスタ (135/500/102/101) と税区分マスタ (T10) を提供済みであること
  - M2 `journal` の仕訳エンジン (`POST /api/journal-entries`, バランス検証込み) が稼働済みであること
  - 取引先は仕訳明細の補助科目 (`sub_account`) として文字列で保持する (M1 取引先マスタ導入前は自由入力)
  - 発生主義の適用対象は役務提供売上のみ。商品販売売上は本モジュールの対象外

- **リスク**:
  - 取引先マスタ (M1-F205) が未確立の段階では補助科目が自由入力文字列となり、表記揺れによる集計誤りのリスクがある。M1 確立後に取引先IDへ移行する必要がある (TBD)
  - 消費税の端数計算方針 (切り捨て/切り上げ/四捨五入) が全体設計で「切り捨て」と定義されているが、取引先との契約により異なる場合の対応は未定 (TBD)

## 7. 未決事項 (TBD)

- [ ] 取引先マスタ (M1-F205) 確立前の暫定対応: 補助科目を自由入力文字列とするか、簡易取引先リストを本モジュールで持つか
- [ ] 入金の口座振替 (銀行引落) と通常振込を区別して記録する要否 (M5 `cash-bank` との連携方針)
- [ ] 分割入金 (1件の売掛に対して複数回の入金) の消込表示方針: 残高ベースか明細対応か
