# 要求事項定義書 — data-io (データ入出力・連携)

## 1. 背景・目的

個人事業主の帳簿管理において、データの可搬性・継続性・安全性は極めて重要である。
本モジュール (`data-io`) は以下の課題を解決する。

1. **記帳効率化 (F-901)**: 仕訳・明細をCSVで一括取込し、手入力コストを削減する。
2. **税理士共有・監査対応 (F-902)**: 仕訳帳・試算表・決算書をCSV/PDFで出力し、税理士への書類共有やバックオフィス連携を実現する（標準形式によるロックイン回避: N-8, Q-7）。
3. **消失防止 (F-903)**: 全データの定期バックアップと手動エクスポート・復元を提供し、データ消失ゼロを担保する（N-2, N-3）。
4. **年度継続性 (F-904)**: 暦年末に期末残高を翌年の期首残高へ繰り越し、複数年にわたる連続した帳簿管理を実現する（P-7）。

## 2. スコープ

- **対象**:
  - 仕訳CSVの一括インポート（列定義・バリデーション含む）
  - 仕訳帳・総勘定元帳・試算表・PL・BS・青色申告決算書のCSVエクスポート
  - PDFエクスポート（将来導入：当面はCSVのみ）
  - 全エンティティ（マスタ・仕訳・固定資産・証憑メタ）のJSON/CSVバックアップと復元
  - 日次自動バックアップ（N-3）
  - 年度繰越（資産・負債・純資産の期末残高を翌期首へ移行 + 損益振替）
- **対象外**:
  - 銀行CSV明細の取込（フォーマット変換・科目推測を伴うものは M5 `cash-bank` 担当）
  - e-Tax 電子申告への直接連携（当面はCSV/PDF出力で代替）
  - 他社会計ソフトへのデータ移行（フォーマット変換アダプタは TBD）
  - 外部クラウドストレージへの自動アップロード（TBD）

## 3. ステークホルダー

| 役割 | 関心事 |
|------|--------|
| 個人事業主 | バックアップがあれば PC 故障でもデータを復元できること。年度が変わっても同じシステムで継続できること |
| 税理士 | 仕訳帳・試算表・決算書を標準的なCSV/PDFで受け取れること。独自フォーマットへの依存を避けたい |
| 開発者（M9担当） | M7 `reports` の帳簿集計 API・M1〜M6 のエンティティに依存し、M8 `closing-tax` の決算書出力と結合すること |
| 開発者（全体） | バックアップ復元テストが CI で自動実行されること |

## 5. 制約条件

- **技術スタック**: FastAPI (Python 3.14) / Vue 3 + Vite + TypeScript（既存スターターに準拠）
- **金額の単位**: 円単位の整数のみ（`JournalLine.amount: int`・`OpeningBalance`に準拠）。小数点以下は扱わない
- **仕訳検証**: CSVインポートで生成する仕訳の借貸バランス検証は、既存の `JournalEntry` モデルの `model_validator` に委譲する。独自検証ロジックを重複実装しない
- **エンティティの定義**: 仕訳・勘定科目・税区分は M1 `core-masters`・M2 `journal` のドメインモデルをそのまま参照する
- **報告書生成**: 帳簿の集計ロジックは M7 `reports`、決算書の集計は M8 `closing-tax` に依存する。data-io は集計を再実装せず、それらの出力をファイル化する役割に限定する
- **PDFエクスポート**: 当面実装しない。将来の導入に備えてAPIパスを確保するにとどめる
- **暦年固定**: 会計年度は1月1日〜12月31日の暦年（P-7）

## 6. 前提・リスク

- **前提**:
  - M1 `core-masters`（勘定科目・税区分・期首残高 API）が提供されていること
  - M2 `journal`（仕訳 CRUD API・バランス検証）が提供されていること
  - M7 `reports`（試算表・PL・BS の集計 API）が提供されていること
  - M8 `closing-tax`（青色申告決算書の集計 API）が提供されていること
  - 元入金（科目コード 400）・事業主貸（410）・事業主借（420）がマスタに存在すること（`store.py` にて確認済み）
  - 収益区分（`AccountType.REVENUE`）・費用区分（`AccountType.EXPENSE`）の科目は年度繰越時にゼロリセットとなること
- **リスク**:
  - M7/M8 のエクスポート用 API が未確定の場合、エクスポート処理の実装が遅延する（`TBD`）
  - 大規模データ（5万件超）でのバックアップ性能が REQ-106 を満たさない可能性がある。永続化（`add-persistence` 導入後）に再測定が必要
  - PDFエクスポートのライブラリ選定（ReportLab / WeasyPrint 等）は未確定（全体設計§9 に合わせ TBD）
  - 日次自動バックアップのスケジューラ実装方式（APScheduler / OS cron / ジョブキュー）は永続化導入後に確定
