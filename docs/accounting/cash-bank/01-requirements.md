# 要求事項定義書 — cash-bank (資金管理)

## 1. 背景・目的

個人事業主は日々の現金・預金の入出金を把握することが帳簿管理の基本であり、
青色申告の要件でもある。現金出納帳・預金出納帳は「いつ・何のために・いくら動いたか」
を時系列で可視化し、残高の正確性を確認する帳簿として機能する。

また、銀行やクレジットカードのCSV明細を取り込んで仕訳候補に変換する機能は、
記帳漏れを防ぎ、簿記非専門のユーザーでも正確な帳簿を維持することを支援する。

本モジュール (`cash-bank`) は M2 `journal` が保持する仕訳ストアを読み取ることで
出納帳ビューを導出し、簡易入力・CSV取込・消込の入口として機能する。

## 2. スコープ

- **対象**:
  - 現金出納帳（勘定科目コード 101: 現金）の表示
  - 普通預金出納帳（勘定科目コード 102: 普通預金など資産科目）の表示
  - 出納帳形式の簡易入力（相手科目1つ＋現金/預金の2明細仕訳に変換）
  - 銀行・クレジットカードCSV明細の取込と仕訳候補の生成
  - 仕訳候補の確定（消込・名寄せ）
- **対象外**:
  - 売掛金・買掛金の消込（M3 `receivables` / M4 `expenses` の担当）
  - 外貨建て取引・為替差損益
  - API連携による銀行明細のリアルタイム取得（将来の TBD）
  - 複数銀行口座の統合残高ビュー（単一口座選択で対応。統合は TBD）

## 3. ステークホルダー

| 役割 | 関心事 |
|------|--------|
| 個人事業主（簿記非専門） | 日々の現金・預金残高をひと目で確認したい。難しい複式簿記を意識せず入力できること |
| 個人事業主（帳簿確認者） | 出納帳の摘要・入出金・残高が正確であること。青色申告に対応できること |
| 税理士（将来） | 仕訳データとの整合性が担保されていること。CSV出力できること |
| 開発者（M2担当との並列） | 仕訳ストアの読み取りAPIが確定すれば出納帳ビューを独立開発できること |

## 5. 制約条件

- **技術スタック**: FastAPI (Python 3.14) / Vue 3 + Vite + TypeScript（既存スターターに準拠）
- **金額の単位**: 円単位の整数。補助単位（銭）は扱わない（既存 `JournalLine.amount: int` に合わせる）
- **仕訳エンジン**: 仕訳の生成・バランス検証は M2 `journal` の `JournalEntry`/`JournalLine` ドメインモデルに委譲する。cash-bank 独自の仕訳モデルは作らない
- **勘定科目マスタ**: M1 `core-masters` の `AccountingStore.accounts` を参照する。独自のマスタは持たない
- **期首残高**: M1 `core-masters` が管理する `OpeningBalance` を参照する（M1が未実装の場合は0円として動作）
- **科目推測（F-105）**: 摘要からの科目自動推測は M2/M5 連携機能。初期実装では摘要キーワードの簡易マッチングで代替し、本格的な推測は TBD

## 6. 前提・リスク

- **前提**:
  - M2 `journal` の仕訳 CRUD API（`GET /api/journal-entries`、`POST /api/journal-entries`）が先行して提供されている
  - M1 `core-masters` の勘定科目一覧 API（`GET /api/accounts`）が利用可能
  - 現金(101)・普通預金(102) の勘定科目が `AccountingStore` に存在する（現状コードで確認済み）
  - 会計年度は暦年固定（1月〜12月）
- **リスク**:
  - 銀行CSVフォーマットは金融機関ごとに異なり、アダプタの追加コストが未定（全体§9参照）
  - 科目の自動推測（F-105）の精度が低い場合、ユーザーの手修正負荷が増える
  - 期首残高API（M1）が未完成の場合、出納帳残高の積み上げが不正確になる（0円起点で暫定動作）
