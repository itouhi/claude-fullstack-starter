# 要求事項定義書 — reports (帳簿・レポート)

## 1. 背景・目的

個人事業主が日々の記帳(仕訳)から各種帳簿・決算資料を即座に参照できるようにする。現状、仕訳データは仕訳ストア (M2) に蓄積されているが、その内容を科目別・期間別・帳票別に集計・表示する機能が必要である。本モジュール (M7) は仕訳ストアを**読み取り専用**で参照し、総勘定元帳・試算表・損益計算書 (PL)・貸借対照表 (BS)・月別売上集計・ダッシュボードを提供する。青色申告決算書 (M8) の集計元データとしても機能する。

由来: 要件定義書 F-701〜F-707 / 全体基本設計 §1.3 (読み取りビュー) / §6.1 (並列開発DAG)

## 2. スコープ

- **対象**:
  - F-701 仕訳帳の閲覧 (実装済み)
  - F-702 総勘定元帳: 科目別の時系列仕訳・相手科目・借貸・繰越残高の積上
  - F-703 試算表 (合計残高試算表) (実装済み)
  - F-704 損益計算書 (PL): 期間損益。収益合計−費用合計=当期純利益
  - F-705 貸借対照表 (BS): 期末時点の財政状態。資産=負債+純資産+当期純利益
  - F-706 ダッシュボード: 売上・利益・資金残高・未入金の可視化 (推奨)
  - F-707 月別売上集計: 売上高を月別に集計 (青色申告決算書 M8 月別欄の元データ)
- **対象外**:
  - 仕訳の登録・訂正・削除 (M2 の責務)
  - 青色申告決算書の帳票出力・PDF生成 (M8 の責務)
  - 消費税の集計・申告 (M8 の責務)
  - 決算整理仕訳の生成 (M8 の責務)
  - 固定資産台帳の参照 (M6 の責務)
  - CSV エクスポートの生成処理 (M9 の責務。本モジュールは集計データを提供するのみ)

## 3. ステークホルダー

| 役割 | 関心事 |
|------|--------|
| 個人事業主 (主ユーザー) | 今期の売上・利益が即座にわかること。確定申告に必要な帳票が揃うこと。操作が簡単なこと |
| 税理士・記帳担当 | 試算表・総勘定元帳が標準的な複式簿記の形式で出力されること。PL/BS の数値が仕訳ストアと完全一致すること |
| M8 closing-tax | PL の当期純利益・月別売上集計・BS の純資産構成を集計元データとして参照できること |

## 5. 制約条件

- 集計は仕訳ストア (`store.journal_entries`) の全有効仕訳 (status != DELETED) を対象とする
- 勘定残高の符号規則は全体基本設計 §4.1 に従う: 資産・費用は借方正、負債・純資産・収益は貸方正 (`DEBIT_POSITIVE_TYPES`)
- 会計年度は暦年固定 (1月〜12月、P-7)
- 金額は円単位の整数 (小数・外貨なし、B-2)
- バックエンド: FastAPI (Python 3.14)。フロントエンド: Vue 3 + TypeScript (Composition API + `<script setup>`)
- 試算表 (`GET /api/reports/trial-balance`) は実装済み。新規帳票は同一ルーターに追加する
- 全 API の `/api` prefix は `main.py` で一括付与 (個別ルーターには付けない)

## 6. 前提・リスク

- **前提**:
  - M1 `core-masters` が勘定科目マスタ (AccountType の5区分含む) を提供済みであること
  - M2 `journal` の仕訳エンジンが稼働し、仕訳が `store.journal_entries` に蓄積されていること
  - 現在は in-memory ストア。DB 移行 (`add-persistence`) 後もリポジトリ IF を通じて同一 API を維持する
  - 期首残高 (OpeningBalance) は現時点では仕訳ストアにエントリがないため、BS の繰越計算は仕訳合計のみで行う (期首残高マスタ導入は TBD)

- **リスク**:
  - 大量仕訳 (数万件) 時の集計パフォーマンス: in-memory ではフルスキャンとなり N-1 目標を達成できない可能性がある。DB 移行後に科目別残高の事前集計テーブルで対処する (TBD)
  - BS における当期純利益の表示: PL (収益−費用) で算出した値を BS の純資産へ組み込む処理が必要。勘定科目マスタに「当期純利益」科目が存在しない場合は計算値として付与する (TBD)
