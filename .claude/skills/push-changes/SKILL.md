# push-changes

変更内容を確認し、**同一内容 (関心事) ごとにコミットを分割**して作成し、**最後にまとめて push** する。

## 入力として確認すること

1. **対象範囲** — 作業ツリー全体か、特定ファイルのみか (デフォルト: 全変更)
2. **ブランチ** — 現在のブランチに積むか、新規ブランチを切るか
   - 現在が `main` (デフォルトブランチ) の場合は **必ず新規ブランチを切る**。ブランチ名をユーザーに確認 (例: `feat/users-list`)

## 手順

### 1. 変更内容を確認する
```
git status
git diff            # ステージ前の変更
git diff --staged   # 既にステージ済みがあれば
```
- 変更されたファイルと差分の中身を実際に読み、**論理的な関心事 (concern) ごとにグルーピング**する
  - 例: 「API 追加」「フロントの表示修正」「lint 設定変更」「ドキュメント追加」は別コミット
  - 同じ目的の変更 (例: 1 機能のための backend + frontend) は 1 コミットにまとめてよい

### 2. ブランチを準備する
```
git rev-parse --abbrev-ref HEAD   # 現在のブランチを確認
```
- `main` 上なら新規ブランチを作成: `git switch -c <branch>`
- 既にフィーチャーブランチ上ならそのまま使う

### 3. グループごとにコミットする (同一内容ごと)
グループの数だけ以下を繰り返す。**関係ないファイルを巻き込まない**よう `git add` はパス指定で行う。
```
git add <そのグループに属するファイルのみ>
git commit -m "<type>(<scope>): <要約>"
```
- メッセージは **Conventional Commits** に揃える (リポジトリの既存履歴に準拠)
  - `feat` / `fix` / `chore` / `docs` / `refactor` / `test` / `style` など
  - 例: `feat(users): ユーザー一覧 API と画面を追加`
- 1 コミット = 1 つの関心事。混在させない
- コミットメッセージ末尾に必ず Co-Authored-By を付与する。モデル名は**実行中のモデル名**を使う (ハードコードしない):
  ```

  Co-Authored-By: <実行中のモデル名> <noreply@anthropic.com>
  ```

### 4. まとめて push する (1 回)
全コミットを作り終えてから **一度だけ** push する。
```
git push -u origin <branch>     # 新規ブランチ / 未追跡の場合
git push                        # 既に upstream 設定済みの場合
```
- push はグループごとに分けず、最後にまとめて 1 回実行する

### 5. 報告
- 作成したコミット一覧 (`git log --oneline -n <件数>`) と push 先ブランチを報告する
- リモートがあれば PR 作成リンクの有無も伝える (PR 自体は依頼があれば別途)

## ガードレール

- **コミットは分割、push はまとめて 1 回** — これが本スキルの主目的
- `git add -A` / `git add .` で無差別にステージしない。グループごとにパス指定する
- `main` などデフォルトブランチへ直接 commit / push しない。必ずブランチを切る
- ユーザーが明示的に push を依頼したときのみ push する (このスキルの起動自体が依頼に相当)
- `--force` 系は使わない (ユーザーが明示的に指示した場合を除く)
- コミット前に差分を必ず確認し、意図しない変更 (デバッグコード・秘密情報) が混ざっていないか点検する
- 機微情報 (`.env`, 鍵, トークン) をコミットしない
