# write-dev-docs

開発プロセスの各工程の成果物ドキュメントを作成するスキル。
工程ごとのテンプレート (`templates/`) を読み込み、`docs/<機能名>/` 配下に Markdown で出力する。

## 工程一覧とテンプレート

| # | 工程 | 出力ファイル | テンプレート | ID 接頭辞 |
|---|------|------------|------------|----------|
| 01 | 要求事項 | `01-requirements.md` | `templates/01-requirements.md` | `REQ-` |
| 02 | 仕様書 | `02-spec.md` | `templates/02-spec.md` | `SPEC-` |
| 03 | 基本設計 | `03-basic-design.md` | `templates/03-basic-design.md` | `BD-` |
| 04 | 詳細設計 | `04-detailed-design.md` | `templates/04-detailed-design.md` | `DD-` |
| 05 | 実装内容 (テストコード含む) | `05-implementation.md` | `templates/05-implementation.md` | `IMP-` (実装) / `TST-` (テストコード) |
| 06 | テスト実施 (テスト計画) | `06-test-plan.md` | `templates/06-test-plan.md` | `TC-` |
| 07 | テスト結果 | `07-test-results.md` | `templates/07-test-results.md` | `TR-` |
| 08 | リリース | `08-release.md` | `templates/08-release.md` | `REL-` |

## 入力として確認すること

1. **機能名** — kebab-case (例: `users-list`, `task-create`)。`docs/<機能名>/` のディレクトリ名になる
2. **対象工程** — 上表の # または工程名。複数指定可 (例: 「要求事項と仕様書」)
3. **入力情報** — その工程を書くための素材 (要件の箇条書き、API 仕様、設計方針、テスト結果ログなど)
   - 不足している場合は推測で埋めず、ユーザーに確認する

## 手順

1. `docs/<機能名>/` が無ければ作成する
2. 対象工程の **テンプレートファイルを Read** する (`templates/<file>`)
3. **前工程の成果物を確認する** (トレーサビリティの起点)
   - `docs/<機能名>/` 内の既存ドキュメントを読み、参照すべき ID (REQ-xxx など) を把握する
   - 前工程が未作成なら、その旨をユーザーに伝える (先に前工程を書くか、リンクを TBD にするか確認)
4. テンプレートの各セクションを、ユーザー提供の入力情報で埋める
   - **空セクションを温存しない**。不要な節は削除、不明な箇所は `TBD` と明記して箇条書きで残課題化
5. **ID を採番する** — 工程の接頭辞 + 連番 (例: `REQ-001`)。各項目に付与し、後工程から参照できるようにする
6. **トレーサビリティ欄を埋める** — その項目がどの前工程 ID に由来するかを必ず記載 (詳細は下記)
7. `docs/<機能名>/README.md` (索引) を作成 or 更新し、作成済み工程へのリンク一覧を載せる
8. **変更履歴を更新する** — 各工程ドキュメント末尾の「変更履歴」表に、今回の作成/更新の**概要**を1行追記する
   - 新規作成は `初版作成`、更新は変更点の要約 (例: `非機能要求 REQ-102 を追加`)
   - 日付は実際の作業日、担当は作業者名。版番号は追記ごとにインクリメント (1.0 → 1.1 …)
   - **過去行は消さない (追記のみ)**
9. 出力した Markdown のパスをユーザーに報告する

## トレーサビリティ (必須)

各ドキュメントは前工程の成果物に紐づける。これにより「この実装は何の要求由来か」を追える。

- **連鎖**: REQ → SPEC → BD → DD → IMP → TC → TR → REL
  - 工程05のテストコードには `TST-` を採番し、`DD-`/`SPEC-` を由来に持つ。後工程のテスト計画 (`TC-`) はこの `TST-` を自動テストの対応先として参照する
- 各項目テーブルに **「由来」列** を設け、上流 ID を列挙する (例: 詳細設計 `DD-003` の由来に `BD-001`)
- ドキュメント冒頭の frontmatter に前工程ファイルへの相対リンクを記載する:

```markdown
---
feature: users-list
phase: 04-detailed-design
upstream:
  - ./03-basic-design.md
---
```

- 後工程を書くときは、対応する上流 ID が実在するか確認する。なければユーザーに不足を報告する

## ガードレール

- ドキュメントは**日本語**で書く (frontmatter のキーは英語)
- 保存先は必ず `docs/<機能名>/`。番号接頭辞 (`01-`〜`08-`) を変えない
- テンプレートの体裁を勝手に簡略化せず、不要節は削除・不明点は `TBD` で残す
- **推測で要件や結果を捏造しない**。特にテスト結果 (07) とリリース (08) は実際の実行ログ・コミット情報に基づく
- 1 ドキュメントずつ完結させる。複数工程を一度に頼まれた場合も工程順 (番号昇順) に作る
- コードの実装そのものは行わない。実装は `add-api-endpoint` / `add-vue-component` / `add-fullstack-feature` スキルの担当。本スキルはその**内容を文書化**する
- 各工程ドキュメントは末尾に「変更履歴」表を持つ。作成・更新のたびに概要を1行**追記**し (削除しない)、いつ・誰が・何を変えたかを追えるようにする。索引 (`README.md`) は常に最新の一覧を反映する性質のため変更履歴は持たせない
