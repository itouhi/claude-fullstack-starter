# run-dev-cycle

機能の開発を、工程ドキュメント・実装・テストまで含めて**逐次パイプライン**で回す。
単発の文書生成ではなく、`write-dev-docs` / `add-*` / テスト実行を束ねる**オーケストレータ**。

> ⚠️ これは**ガイド付き手動オーケストレーション**であり、自動実行エンジンではない。各工程は Claude が
> 本スキルの手順に従って実施する。価値は「抜け漏れ防止・一貫した成果物・トレーサビリティ維持」にある。

## 入力として確認すること
1. **機能名** — kebab-case (`docs/<機能名>/` に対応)
2. **対象範囲** — どの工程からどこまで (例: `03→07`, `01→08`)
3. **自動度** — ゲートあり半自動 (推奨) / 全自動 / 工程ごと手動
4. **実験ブランチか** — 検証用なら `sandbox` (PR禁止) で回す

## 進行管理
- 真実の情報源は `docs/<機能名>/README.md` の状態列。
- **1ティック = 1工程**。未完の最小番号工程を特定して実行する。

## ティック処理
1. README 索引から未完の最小番号工程 N を特定
2. 上流 `N-1` が `done`・承認済みか確認 → 未充足なら停止して報告
3. 工程 N を実行 (工程の性質で使うスキルが変わる):
   | 工程 | 実作業 | 使用 |
   |------|--------|------|
   | 01 要求/02 仕様/03 基本設計/04 詳細設計/06 テスト計画 | 文書化 | `write-dev-docs` |
   | 05 実装 | **実コード実装** + テストコード → 文書化 | `add-api-endpoint` / `add-vue-component` / `add-fullstack-feature` → `write-dev-docs` |
   | 07 テスト結果 | **実テスト実行** (pytest 等) → 結果記録。UI は `verifier-webapp` | (実行) → `write-dev-docs` |
   | 08 リリース | 実コミット/タグ/手順に基づき記録 | `write-dev-docs` |
4. 索引・変更履歴・トレーサビリティ (由来ID連鎖) を更新
5. **ゲート判定** → 該当なら停止、なければ次ティック

## 工程ごとの成果物チェックリスト
各ティックの完了は、以下が揃ったかで判定する。
- [ ] 該当工程の `docs/<機能名>/0N-*.md` を作成 (空節は削除、不明点は TBD)
- [ ] 由来 ID を採番し、上流 ID への「由来」列を記入 (トレーサビリティ連鎖)
- [ ] frontmatter の `upstream` を前工程ファイルに設定
- [ ] 末尾の「変更履歴」に概要を追記
- [ ] `docs/<機能名>/README.md` の状態列を更新
- [ ] (05) 実コード + テストが追加され、ruff/型/テストが通る
- [ ] (07) 実テスト実行ログを記録、UI は `verifier-webapp` で確認

## ゲート (停止点)
- 🚦 **04 詳細設計の完了後** (実装着手の直前) — 設計承認
- 🚦 **テスト FAIL 時** — 即停止 (捏造せず失敗を記録)
- 🚦 **対象範囲の最終工程の完了後** — 完了報告

## 機構 (任意)
- `/loop` 動的ペースで 1ティックずつ自走させ、ゲートで一時停止できる。
- 工程間に外部待ちは無いため、セグメント内のティックは連続実行してよい (スリープしない)。停止は必ずゲートで行う。

## ガードレール
- **07/08 は実行ログ・実コミットに基づく。結果を捏造しない**。FAIL は FAIL と記録。
- 実装は必ず `add-*` スキルの規約 (CLAUDE.md: `response_model` 必須 / `<script setup lang="ts">` / API通信は `services/` 集約 / pytest 最低1件) に従う。
- 実装の命名・仕様コメント (docstring/TSDoc・工程ID 紐づけ)・フォーマットは `coding-standards` スキルに従う。
- 05 実装後 (テスト前後) に `/code-review` でレビューし、指摘を解消してから次工程・完了に進む (品質関門)。
- テストは backend=pytest、frontend 単体=`add-frontend-test` (Vitest)、回帰E2E=`add-e2e-test` (Playwright)。基盤が要れば `add-persistence`(DB)/`add-auth`(認証) を併用。
- 検証目的の回し方は `sandbox` で行い、成果を main に流さない。本番フローは main からフィーチャーブランチで。
- 関連スキル: `write-dev-docs` (文書化) / `verifier-webapp` (UI 検証) / `push-changes` (コミット&push)。
